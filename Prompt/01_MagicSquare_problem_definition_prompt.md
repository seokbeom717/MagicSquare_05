# STEP 1 — Observation (관찰)

---

## 1. 현재 우리가 해결하려는 상황은 무엇인가?

1부터 16까지의 서로 다른 정수를 4×4 격자에 배치할 때, **어떤 배치가 "균형 조건"을 만족하는가**를 판별하고 생성하는 상황이다.

구체적으로 관찰되는 상황:

- 16개의 숫자가 존재하며, 이들을 16개의 칸에 중복 없이 채워야 한다
- 배치의 경우의 수는 16! ≈ 2조 개 이상으로 방대하다
- 그 중 **모든 행, 열, 대각선의 합이 동일한 값(34)** 이 되는 배치만이 유효하다
- 유효한 배치를 찾는 기준이 명확하지만, 탐색 공간이 극단적으로 넓다

즉, 현재 상황은 **"제약 조건을 만족하는 배치를 방대한 공간에서 효율적으로 탐색해야 하는 문제"** 다.

---

## 2. 왜 4×4 마방진 문제를 다루는가?

4×4 마방진은 아래의 이유로 학습과 설계 양쪽에서 특별한 위치를 갖는다.

| 관찰 항목 | 내용 |
|---|---|
| 크기의 적절성 | 3×3은 너무 단순(해가 1가지), 5×5 이상은 너무 복잡 — 4×4는 **복잡성과 가시성의 균형점** |
| 해의 다양성 | 4×4 마방진의 본질적으로 다른 해는 880가지 — 단일 정답이 아니라 **탐색과 검증의 대상** |
| 제약의 명확성 | 합의 조건(34)이 수학적으로 명확 — 정답 여부를 즉시 검증할 수 있음 |
| 역사적 맥락 | 뒤러의 "멜랑콜리아"(1514)에 등장 — 수학적 미학과 완결성의 상징 |

따라서 4×4는 **"탐색 알고리즘 학습에 이상적인 크기"** 로 선택된 것으로 관찰된다.

---

## 3. 어떤 학습 또는 시스템 설계 맥락에서 등장하는가?

이 문제는 다음 맥락들이 **교차하는 지점**에 위치한다:

**알고리즘 학습 맥락**
- 완전 탐색(Brute Force)과 그 한계 인식
- 백트래킹(Backtracking)을 통한 가지치기(pruning) 전략
- 제약 충족 문제(CSP, Constraint Satisfaction Problem)의 실습 사례

**시스템 설계 맥락**
- 입력 → 검증 → 출력의 명확한 파이프라인
- 하나의 해 vs. 모든 해 탐색이라는 **목적의 선택** 이 설계에 영향을 줌
- 시각적 출력(4×4 격자 표시)이 요구되는 UI/출력 설계 문제

**문제 정의 맥락**
- "만든다"는 행위 이면에는 **탐색, 검증, 선택, 표현** 이라는 4개의 하위 문제가 숨어 있음
- 이 네 가지를 어떻게 분리하고 설계하느냐가 핵심 설계 결정

---

**관찰 요약:**

> 4×4 마방진 문제는 "숫자를 채운다"는 단순한 행위처럼 보이지만, 실제로는 **방대한 탐색 공간에서 다중 제약 조건을 동시에 만족하는 상태를 효율적으로 찾고 표현하는 문제**다. 이 문제의 본질은 배치 자체가 아니라, **탐색 전략과 제약 처리 방식의 선택**에 있다.

# Q: 왜 마방진을 완성해야 하는가?

---

## 방법론: 가정(A)을 놓고 불편함을 드러내기

하나의 정답을 선언하지 않는다. 가능한 A를 복수로 가정하고, 각 답이 숨기고 있는 **구조적 긴장**을 분석한다.

---

## A-1. "알고리즘을 배우기 위해서"

> 마방진은 백트래킹, 완전 탐색, 제약 충족 문제를 실습하기 좋은 예제이기 때문이다.

### 드러나는 불편함

| 관찰 | 구조적 문제 |
|---|---|
| 마방진 완성이 **목적**처럼 보이지만, 실제 목적은 **알고리즘 이해** | 목적과 수단이 뒤바뀌어 있다 |
| 알고리즘을 배웠다면, 마방진이 "완성"되어야 하는가? | **완성 기준이 모호하다** — 하나의 해? 모든 해? 가장 빠른 해? |
| 마방진 외에도 같은 알고리즘을 배울 수 있는 문제는 많다 | **왜 하필 마방진인가**라는 질문이 해소되지 않는다 |

**구조적 문제:** 학습 목적이라면 완성이 아니라 **탐색 과정 자체**가 결과물이어야 한다. "완성"이라는 단어가 목적을 흐린다.

---

## A-2. "정해진 규칙을 만족하는 결과물을 출력하기 위해서"

> 모든 행, 열, 대각선의 합이 34가 되는 배치를 화면에 표시해야 하기 때문이다.

### 드러나는 불편함

| 관찰 | 구조적 문제 |
|---|---|
| 출력 조건(합=34)은 명확하지만, **어떤 배치를 출력할지**는 결정되지 않았다 | 880가지 해 중 **선택 기준이 없다** |
| "완성"을 출력으로 정의하면, 프로그램은 단순 출력기가 된다 | 탐색 로직과 출력 로직이 **분리되지 않은 채** 설계될 위험 |
| 사용자가 입력을 줄 수 있는가? 없는가? | **입력-처리-출력 경계**가 정의되지 않았다 |

**구조적 문제:** "완성 = 출력"으로 정의하면 시스템이 **정적 생성기**에 머문다. 검증기, 교육 도구, 게임 등 확장 가능성이 처음부터 차단된다.

---

## A-3. "수학적 완결성을 구현으로 증명하기 위해서"

> 마방진은 수학적으로 아름다운 구조를 가지며, 이를 코드로 구현함으로써 그 완결성을 확인한다.

### 드러나는 불편함

| 관찰 | 구조적 문제 |
|---|---|
| "아름다운 구조"는 주관적 판단이다 | **완성의 기준이 수학인가, 코드인가, 사람의 감상인가** — 불명확 |
| 수학적 증명은 이미 되어 있다(880가지 해가 존재함은 알려진 사실) | 구현이 **새로운 지식을 생산하는가**, 아니면 **알려진 사실을 재현하는가** |
| 완결성을 증명하려면 **모든 해를 열거**해야 하지만, 하나만 출력해도 "완성"이라 부를 수 있다 | 완성의 정의가 **부분적 성취**를 전체 성취로 오인하게 만든다 |

**구조적 문제:** 이 답은 **완성이라는 단어를 가장 무겁게 사용**한다. 그러나 그 무게를 지탱할 기준이 설계에 없다.

---

## A-4. "사용자가 요청했기 때문에"

> 과제 또는 요구사항으로 마방진 프로그램 완성이 명시되었기 때문이다.

### 드러나는 불편함

| 관찰 | 구조적 문제 |
|---|---|
| 요청이 있다고 해서 **왜(Why)가 해소**되는 것은 아니다 | Why를 외부에 위임하면, 설계자가 **판단 기준을 잃는다** |
| "완성"의 범위를 요청자가 정의하지 않았다면 | **암묵적 가정**이 설계 전반에 퍼진다 |
| 요청이 바뀌면 프로그램 전체를 바꿔야 하는가? | **변경에 취약한 설계**가 처음부터 예정된다 |

**구조적 문제:** 가장 흔하고 가장 위험한 답이다. Why 없이 What만 존재하면, 설계는 **요청의 노예**가 된다.

---

## 종합 분석

```
A-1 → 목적과 수단이 전도되어 있다
A-2 → 완성의 범위와 선택 기준이 정의되지 않았다
A-3 → 완성의 기준이 수학, 코드, 감상 중 무엇인지 불명확하다
A-4 → Why를 외부에 위임하면 설계 판단 기준이 사라진다
```

**공통적으로 드러나는 구조적 문제:**

> "마방진을 완성해야 한다"는 문장은 **Why가 없는 What**이다.
> 완성이라는 단어 안에 탐색 전략, 해의 선택, 출력 형식, 사용자 목적이 모두 압축되어 있으며,
> 이것이 해소되지 않은 채 설계로 넘어가면 **모든 결정이 암묵적 가정 위에 세워진다.**

---

**다음 단계 제안:**

"완성"이라는 단어를 해체하여 — **탐색 / 검증 / 선택 / 표현** — 네 개의 하위 문제로 분리하고, 각각에 대해 Why를 다시 묻는 것이 필요하다.

# Q: 왜 단순 계산이 아니라 프로그램으로 구현하는가?

---

## 전제: "단순 계산"과 "프로그램" 사이의 간극

손으로 1~16을 배치하여 합이 34가 되는지 확인하는 것은 **가능**하다.
그러나 그것은 **1회적 행위**이며, 다음 배치를 검증하려면 처음부터 다시 시작해야 한다.
이 간극이 프로그램 구현의 필요성을 만든다.

---

## 관점 1 — 반복 가능성 (Repeatability)

**단순 계산의 한계:**

```
시도 1: 특정 배치 → 합 확인 → 실패 → 처음부터
시도 2: 다른 배치 → 합 확인 → 실패 → 처음부터
...
시도 N: 성공 또는 포기
```

16!개의 경우의 수를 사람이 반복하는 것은 물리적으로 불가능하다.

**프로그램이 제공하는 것:**

- 동일한 탐색 로직을 **조건이 충족될 때까지 자동으로 반복**한다
- 반복의 횟수, 속도, 종료 조건을 **코드로 명시**할 수 있다
- 반복 과정 자체가 기록되고 재현된다

**드러나는 구조적 의미:**

> 프로그램은 "사람이 반복할 수 없는 것"을 위임받는 도구다.
> 반복 가능성은 단순 편의가 아니라, **해결 가능성의 조건**이다.

---

## 관점 2 — 검증 자동화 (Verification Automation)

**단순 계산에서의 검증:**

배치가 완성되면 사람이 직접 확인한다.
- 4개 행의 합: 4번 계산
- 4개 열의 합: 4번 계산
- 2개 대각선의 합: 2번 계산
- 총 **10번의 수동 계산**, 매 배치마다 반복

오배치 1개가 섞이면 검증 전체를 다시 해야 한다.

**프로그램이 제공하는 것:**

```
is_valid(grid) → 10개 조건을 즉시 판별 → True / False
```

- 검증 로직이 **코드로 분리**되어 탐색과 독립적으로 존재한다
- 검증은 **탐색 중 매 단계마다 호출** 가능하다 (조기 종료)
- 검증 기준이 바뀌면 **한 곳만 수정**하면 된다

**드러나는 구조적 의미:**

> 자동화된 검증은 단순히 빠른 것이 아니라,
> **탐색과 검증을 분리 가능한 독립적 관심사로 만드는 설계 결정**이다.
> 이 분리가 없으면 탐색 로직이 검증 로직을 내포하게 되어 변경에 취약해진다.

---

## 관점 3 — 오류 방지 (Error Prevention)

**단순 계산에서 발생하는 오류의 유형:**

| 오류 유형 | 발생 원인 | 감지 가능성 |
|---|---|---|
| 숫자 중복 배치 | 손으로 쓰다가 같은 수 두 번 사용 | 검증 후에야 발견 |
| 합산 오류 | 계산 실수 | 즉시 감지 불가 |
| 누락 | 빈 칸을 채우지 않음 | 육안 확인 필요 |
| 범위 오류 | 1~16 외 숫자 사용 | 의식하지 않으면 놓침 |

**프로그램이 제공하는 것:**

- **입력 단계에서** 1~16 범위 및 중복 여부를 즉시 차단
- 배치 생성 로직 자체가 중복을 구조적으로 허용하지 않음
- 오류가 런타임이 아니라 **설계 단계에서 제거**됨

**드러나는 구조적 의미:**

> 프로그램은 오류를 "나중에 발견"하는 것이 아니라
> **오류가 발생할 수 없는 구조를 설계**한다.
> 이것이 방어적 프로그래밍(defensive programming)의 출발점이다.

---

## 관점 4 — 규칙 기반 사고 훈련 (Rule-Based Thinking)

**단순 계산에서의 사고 방식:**

```
"느낌상 이 숫자가 여기 들어갈 것 같다"
"대각선이 안 맞으니 이걸 바꿔보자"
```

이것은 **휴리스틱(heuristic)** 이며, 재현 불가능하고 전달 불가능하다.

**프로그램 구현이 요구하는 사고 방식:**

```
"어떤 조건이 충족되면 이 칸에 이 숫자를 놓을 수 있는가?"
"어떤 조건이 위반되면 탐색을 중단해야 하는가?"
"완료 조건은 무엇인가?"
```

이것은 **규칙의 명시화**다.

| 사고 항목 | 단순 계산 | 프로그램 구현 |
|---|---|---|
| 조건 표현 | 암묵적, 직관적 | 명시적, 형식적 |
| 결정 기준 | 경험 의존 | 로직 의존 |
| 전달 가능성 | 설명하기 어려움 | 코드가 곧 설명 |
| 재현 가능성 | 같은 사람도 매번 다름 | 항상 동일한 결과 |

**드러나는 구조적 의미:**

> 프로그램으로 구현한다는 것은
> **"내가 아는 것"을 "기계가 실행할 수 있는 언어"로 번역하는 훈련**이다.
> 이 번역 과정에서 직관 속에 숨어 있던 모호함이 강제로 드러난다.
> 마방진은 이 훈련의 재료이고, 프로그램은 그 훈련의 형식이다.

---

## 종합

```
반복 가능성  → 해결 가능성의 조건을 확보한다
검증 자동화  → 탐색과 검증을 독립적 관심사로 분리한다
오류 방지    → 오류를 발견하는 것이 아니라 불가능하게 설계한다
규칙 기반 사고 → 직관을 형식으로 번역하는 과정 자체가 목적이다
```

**결론:**

> 프로그램으로 구현하는 이유는 "빠르기 때문"이 아니다.
> **"규칙을 명시하고, 반복을 위임하고, 검증을 분리하고, 오류를 구조적으로 제거하는"**
> 네 가지 행위를 동시에 수행하기 위해서다.
> 이 네 가지는 마방진 문제를 넘어 모든 소프트웨어 설계의 기반이다.

# Q: 왜 이 문제를 TDD 방식으로 설계하려 하는가?

---

## 전제: TDD는 테스트 도구가 아니다

흔한 오해: TDD = "코드를 짜고 테스트를 추가하는 것"
실제 의미: TDD = **"무엇이 올바른가를 먼저 정의하고, 그 정의에 맞게 코드를 수렴시키는 것"**

마방진은 이 접근이 특히 적합한 문제다. 그 이유를 세 관점으로 분석한다.

---

## 관점 1 — 무엇이 통제되어야 하는가?

### 통제 대상의 목록

마방진 프로그램에는 **통제되지 않으면 조용히 틀려버리는** 지점들이 있다.

| 통제 대상 | 통제되지 않을 때 발생하는 일 |
|---|---|
| 숫자의 범위 (1~16) | 17이 들어가도 합이 34가 될 수 있다 |
| 숫자의 중복 여부 | 같은 숫자가 두 번 나와도 합 조건을 통과할 수 있다 |
| 행/열/대각선의 합 | 일부만 34이고 나머지는 아닌 배치가 "완성"으로 취급될 수 있다 |
| 격자의 완전성 | 빈 칸이 있는 상태로 유효하다고 판정될 수 있다 |

**이것이 TDD를 요구하는 이유:**

```
통제 대상이 많고, 각 조건이 독립적으로 위반될 수 있다
→ 각 조건을 개별 테스트로 명시하지 않으면
→ "대략 맞아 보이는" 코드가 조용히 틀린 채로 진행된다
```

TDD는 **"조용한 오류"를 "명시적 실패"로 바꾸는 장치**다.

### 통제의 계층 구조

```
Level 0: 입력 통제    → 유효하지 않은 숫자가 격자에 들어올 수 없다
Level 1: 배치 통제    → 중복 없이 1~16이 정확히 한 번씩 사용되었다
Level 2: 조건 통제    → 행, 열, 대각선의 합이 모두 34다
Level 3: 결과 통제    → 출력이 4×4 형태로 올바르게 표현된다
```

TDD 없이는 이 계층이 **코드 안에 암묵적으로 섞인다.**
TDD를 통해 각 계층이 **독립적으로 검증 가능한 단위**가 된다.

---

## 관점 2 — 어떤 불변 조건이 존재하는가?

### 불변 조건 (Invariant)의 정의

> 불변 조건이란, 프로그램의 어떤 상태에서도 **항상 참이어야 하는 명제**다.
> 이것이 위반되는 순간 프로그램의 결과는 신뢰할 수 없다.

### 마방진의 불변 조건 목록

**수학적 불변 조건:**

```
INV-1: 1부터 16까지의 정수가 정확히 한 번씩 사용된다
INV-2: 모든 행의 합 = 34
INV-3: 모든 열의 합 = 34
INV-4: 주대각선의 합 = 34
INV-5: 반대각선의 합 = 34
```

**구조적 불변 조건:**

```
INV-6: 격자는 정확히 4행 4열이다
INV-7: 모든 칸이 채워져 있다 (None, 0, 공백 없음)
INV-8: 합의 기준값은 n(n²+1)/2 = 34로 고정된다 (n=4일 때)
```

### 이 불변 조건들이 TDD를 요구하는 이유

```python
# 이 함수가 올바른가?
def is_magic(grid):
    return sum(grid[0]) == 34
```

INV-2 하나만 확인한다. INV-1, 3, 4, 5는 침묵한다.
테스트 없이는 이 함수가 **부분적으로만 맞다**는 사실을 알 수 없다.

TDD에서는 불변 조건 하나가 **테스트 하나**에 대응된다.
불변 조건의 수만큼 테스트가 존재하며, 모든 테스트가 통과할 때 비로소 "완성"이 정의된다.

> **불변 조건을 먼저 열거하는 행위 자체가 TDD의 첫 번째 단계다.**

---

## 관점 3 — 왜 입력/출력이 명확해야 하는가?

### 입력/출력이 불명확할 때 발생하는 일

```
함수 A가 리스트를 받는다 → 어떤 리스트? 1D? 2D? 중첩?
함수 B가 결과를 반환한다 → True/False? 배치 자체? 문자열?
```

불명확한 입출력은 **함수 사이의 계약이 없는 상태**다.
계약이 없으면 테스트를 작성할 수 없고, 테스트가 없으면 리팩토링이 불가능하다.

### 마방진에서 입출력을 명확히 정의한다는 것

**검증 함수의 경우:**

```
입력: List[List[int]] — 4×4 중첩 리스트, 정수값
출력: bool — 모든 불변 조건 충족 여부
부작용: 없음 (순수 함수)
```

**생성 함수의 경우:**

```
입력: 없음 (또는 시드값)
출력: List[List[int]] — 유효한 마방진 1개
실패 조건: 해가 없을 경우 → 예외? 빈 리스트? None?
```

**출력 함수의 경우:**

```
입력: List[List[int]]
출력: str 또는 None (화면 출력)
형식: 각 숫자가 정렬된 4행 4열 텍스트
```

### 입출력 명확성이 TDD를 가능하게 하는 이유

```
명확한 입출력
    → 테스트에서 "주어진 입력에 대해 기대하는 출력"을 쓸 수 있다
    → 테스트가 실행 가능해진다
    → 코드가 테스트를 통과하도록 수렴한다
    → 인터페이스가 구현보다 먼저 확정된다
```

반대 방향:

```
불명확한 입출력
    → 테스트를 쓸 수 없다
    → 코드가 먼저 작성된다
    → 인터페이스가 구현에 종속된다
    → 나중에 변경하면 모든 것이 흔들린다
```

> **입출력의 명확성은 TDD의 전제 조건이 아니라, TDD가 만들어내는 결과물이다.**
> 테스트를 먼저 쓰려면 입출력이 명확해야 하고,
> 입출력을 명확히 하려면 테스트를 먼저 생각해야 한다.
> 이 순환이 설계를 정제한다.

---

## 종합

```
통제 대상    → 조용한 오류를 명시적 실패로 전환하기 위해 TDD가 필요하다
불변 조건    → 불변 조건 열거 자체가 TDD의 첫 번째 단계다
입출력 명확성 → TDD가 인터페이스를 구현보다 먼저 확정시킨다
```

**핵심 명제:**

> TDD로 마방진을 설계하는 이유는
> "테스트를 잘 만들기 위해서"가 아니다.
> **"무엇이 올바른가를 코드보다 먼저 언어로 정의하기 위해서"** 다.
>
> 마방진은 불변 조건이 수학적으로 명확하고,
> 입출력 경계가 뚜렷하며,
> 통제 대상이 계층적으로 분리 가능한 문제다.
> 이 세 조건이 모두 충족되는 문제는 TDD의 효과가 가장 극대화된다.

# 4×4 마방진 — 문제 정의 문서

---

## 1. 표면 문제 정의 (잘못된 정의)

> "1부터 16까지의 숫자를 4×4 격자에 배치하여, 모든 행·열·대각선의 합이 34가 되도록 마방진을 완성하는 프로그램을 만든다."

### 왜 이 정의가 잘못되었는가

| 문제 항목 | 내용 |
|---|---|
| **목적의 부재** | "완성"이 무엇인지 정의되지 않았다 |
| **수신자의 부재** | 누구를 위한 프로그램인가가 명시되지 않았다 |
| **완성 기준의 모호함** | 하나의 해인가, 모든 해인가, 특정 해인가 |
| **What만 있고 Why가 없다** | 무엇을 만드는지는 서술하지만, 왜 만드는지는 없다 |
| **수단이 목적으로 둔갑** | 마방진 생성이 목적처럼 보이지만 실제 목적이 아닐 수 있다 |
| **암묵적 가정의 내포** | "배치한다", "완성한다"는 표현 안에 설계 결정이 숨어 있다 |

---

## 2. 개선된 문제 정의 (정확한 정의)

### 상황

16개의 서로 다른 정수(1~16)를 4×4 격자에 배치할 때, 수학적으로 균형 잡힌 배치가 존재하는가를 **판별하고 표현하는 문제**다.

### 목적

- 방대한 탐색 공간에서 **복수의 불변 조건을 동시에 만족하는 상태를 찾는다**
- 찾은 상태를 **명확한 형식으로 표현하고 검증한다**
- 이 과정을 **재현 가능하고 신뢰할 수 있는 방식으로 설계한다**

### 완성의 정의

| 구분 | 내용 |
|---|---|
| **최소 완성** | 불변 조건을 모두 만족하는 배치가 1개 이상 존재함을 확인하고 표현한다 |
| **확장 완성** | 조건을 만족하는 모든 배치를 열거하고 그 수를 확인한다 |
| **검증 완성** | 임의의 배치가 조건을 만족하는지 판별할 수 있다 |

세 가지 완성은 **독립적으로 정의 가능하며, 각각 별도의 검증 대상이다.**

### 수신자

이 문제를 다루는 사람은 마방진 자체보다 **문제를 구조화하고 조건을 형식화하는 사고 과정**을 훈련한다.

---

## 3. 이 문제의 핵심 Invariant

> Invariant란, 어떤 상태에서도 반드시 참이어야 하는 조건이다.
> 이것이 위반되는 순간 결과는 유효하지 않다.

### 수학적 Invariant

| ID | 조건 | 설명 |
|---|---|---|
| **INV-M1** | 사용된 정수의 집합 = {1, 2, 3, ..., 16} | 범위를 벗어나거나 빠진 수가 없다 |
| **INV-M2** | 모든 정수는 정확히 한 번만 사용된다 | 중복 없음, 누락 없음 |
| **INV-M3** | 4개 행 각각의 합 = 34 | 행 조건 |
| **INV-M4** | 4개 열 각각의 합 = 34 | 열 조건 |
| **INV-M5** | 주대각선의 합 = 34 | 좌상→우하 |
| **INV-M6** | 반대각선의 합 = 34 | 우상→좌하 |

### 구조적 Invariant

| ID | 조건 | 설명 |
|---|---|---|
| **INV-S1** | 격자는 정확히 4행 4열이다 | 형태 불변 |
| **INV-S2** | 모든 칸이 채워져 있다 | 공백, 미정의 값 없음 |
| **INV-S3** | 기준 합 = n(n²+1)/2, n=4일 때 34 | 합의 유일성 보장 |

### Invariant 간의 관계

```
INV-M1 + INV-M2 → 배치의 유효성 (누가 들어왔는가)
INV-M3 ~ INV-M6 → 균형의 유효성 (어떻게 배치되었는가)
INV-S1 + INV-S2 → 형태의 유효성 (구조가 온전한가)

세 종류가 모두 충족될 때만 "유효한 마방진"이다.
하나라도 위반되면 나머지 조건의 충족 여부와 관계없이 무효다.
```

---

## 4. 우리가 실제로 훈련하려는 사고 능력

### 4-1. 조건의 형식화 능력

> 직관적으로 "맞다"고 느끼는 것을 **명제로 표현**할 수 있는가.

- "균형 잡혀 있다"는 감각을 → 수치 조건으로 번역한다
- 모호한 언어를 → 참/거짓으로 판별 가능한 문장으로 바꾼다
- 이 훈련 없이는 조건이 코드 안에 암묵적으로 숨는다

---

### 4-2. 문제 분해 능력

> 하나의 큰 문제를 **독립적으로 검증 가능한 단위로 쪼개는** 능력.

- "마방진을 만든다"는 하나의 행위가 아니라 **탐색 / 검증 / 선택 / 표현** 의 결합이다
- 각 단위가 분리되어야 하나가 바뀌어도 나머지가 안전하다
- 분해되지 않은 문제는 테스트할 수 없고, 테스트할 수 없으면 신뢰할 수 없다

---

### 4-3. 불변 조건 우선 사고

> 구현을 생각하기 전에 **"무엇이 항상 참이어야 하는가"를 먼저 묻는** 습관.

- 불변 조건은 설계의 뼈대다
- 뼈대가 없으면 구현이 먼저 완성되고, 조건은 나중에 끼워 맞춰진다
- 끼워 맞춰진 조건은 예외 상황에서 반드시 무너진다

---

### 4-4. Why → What → How 순서 사고

> 왜 하는가를 먼저 정의하고, 무엇을 만드는가를 결정한 뒤, 어떻게 만드는가를 마지막에 결정하는 순서.

```
Why  : 제약 조건을 만족하는 상태를 재현 가능하게 찾고 검증한다
What : 생성기, 검증기, 표현기의 세 역할
How  : (이 단계는 아직이다)
```

이 순서가 역전되면 — How가 Why를 결정하게 되면 —
구현이 목적을 지배하고, 목적은 구현에 종속된다.

---

### 4-5. 경계 인식 능력

> 문제가 **어디서 시작하고 어디서 끝나는가**를 명확히 긋는 능력.

| 경계 항목 | 이 문제의 경계 |
|---|---|
| 입력 경계 | 1~16, 중복 없음, 4×4 |
| 출력 경계 | 유효한 배치의 표현, 또는 유효/무효 판정 |
| 책임 경계 | 이 문제는 "어떤 배치가 유효한가"를 다루며, "어떤 배치가 더 좋은가"는 다루지 않는다 |
| 확장 경계 | n×n으로의 일반화는 이 문제의 범위 밖이다 |

## 요약

```
표면 정의   → What만 있고 Why가 없으며 완성 기준이 모호하다
개선된 정의 → 탐색·검증·표현의 분리와 세 가지 완성 기준을 명시한다
핵심 Invariant → 수학적(M1~M6) + 구조적(S1~S3), 모두 충족 시에만 유효하다
훈련 목표  → 형식화·분해·불변 조건 우선·Why→What→How·경계 인식
```

> 이 문서가 말하는 것:
> **마방진을 만드는 것이 목적이 아니다.**
> 마방진이라는 재료를 통해, 조건을 먼저 정의하고 구현을 나중에 결정하는 사고 방식을 훈련하는 것이 목적이다.
