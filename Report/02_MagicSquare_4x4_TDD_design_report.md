# Magic Square 4x4 TDD 설계 문서

## 1. 문제 정의
Magic Square 4x4의 목표는 1~16 정수를 중복 없이 정확히 한 번씩 사용해 4x4 보드를 구성하고, 모든 행/열/주대각선(좌상↘우하, 우상↙좌하)의 합이 동일한 상태를 만족하는 해를 생성 또는 검증하는 것이다. 본 문서는 TDD 방식으로 요구사항을 테스트 가능한 단위로 분해하고, 실패 테스트부터 시작해 최소 구현과 리팩터링으로 점진적으로 완성하는 설계 기준을 정의한다.

- 입력
  - `generate()` 기준: 없음(또는 옵션 객체)
  - `validate(board)` 기준: 4x4 정수 보드
- 출력
  - `generate()`: 유효한 4x4 마방진 1개(또는 정책에 따라 복수/첫 해)
  - `validate(board)`: 유효성 결과(`valid: boolean`, 실패 사유 코드 목록)
- 제약사항
  - 값 범위: 1~16
  - 원소 수: 16개
  - 중복 금지
  - 행/열/양 대각선 합 동일
  - 부동소수점, 문자열 등 비정수 타입 불허
- 성공 조건(Definition of Done)
  - 요구사항 ID별 P0 테스트 100% 통과
  - 설계된 P1/P2 테스트 중 범위 내 항목 통과
  - `generate()` 산출물은 `validate()`에서 항상 `valid=true`
  - 회귀 테스트 실행 시 기존 통과 항목 실패 없음
  - 문서의 요구사항-테스트 매핑이 누락 없이 유지됨

## 2. 도메인 모델 및 용어
- 셀(Cell): 보드의 단일 위치 `(row, col)`에 배치되는 값.
- 보드(Board): `4x4` 정수 매트릭스.
- 후보군(Candidates): 아직 사용하지 않은 값 집합.
- 제약(Constraint): 범위, 유일성, 합 일치 등 검증 규칙.
- 부분 상태(Partial State): 일부 셀만 채워진 중간 탐색 상태.
- 목표 합(Magic Constant): 4x4에서 `34` (1~16 합 136 / 4).

데이터 표현 방식:
- 보드 타입: `number[4][4]` 또는 길이 16의 1차원 배열 + 인덱스 변환.
- 좌표 체계: 0-based 인덱스 `(0..3, 0..3)`.
- 상태 스냅샷:
  - `board`
  - `usedNumbers` (Set)
  - `rowSums[4], colSums[4], diagMain, diagAnti`
  - `nextCellIndex` (탐색 순서)
- 결과 모델
  - `ValidationResult { valid, errors[] }`
  - `GenerateResult { board, stats(optional) }`

## 3. TDD 전략
Red-Green-Refactor를 기능 축으로 적용한다. 먼저 "보드 형식 검증" 같은 가장 작은 실패 테스트를 만들고(Red), 테스트를 통과시키는 최소 로직만 구현한 뒤(Green), 공통 검증기/에러코드 정리로 중복을 제거한다(Refactor). 이후 "유일성", "합 제약", "탐색 완성" 순으로 제약을 누적한다.

- 테스트 진화 계획
  1. 입력 형식/크기/타입 검증
  2. 값 범위 및 중복 검증
  3. 행/열/대각선 합 검증
  4. 부분 상태 제약 점검(탐색 중 가지치기)
  5. 생성기 완성 및 end-to-end 검증
- 테스트 레이어
  - Unit(주력): 제약 함수, 합 계산, 후보군 갱신, 가지치기 판정
  - Integration(보조): `generate -> validate` 흐름
  - E2E(최소): 생성 결과가 요구조건을 반복적으로 만족하는지 샘플 실행
- 리팩터링 원칙
  - 실패 사유 코드 표준화
  - 순수 함수 우선(입력 -> 출력)
  - 탐색 상태 변경은 명시적(복사 또는 엄격한 백트래킹 규칙)

- RQ-02(범위/유일성): T-UNI-001, T-UNI-002
- RQ-03(합 제약): T-SUM-001, T-SUM-002, T-SUM-003
- RQ-04(생성 가능성): T-GEN-001, T-GEN-002
- RQ-05(오류 견고성): T-ERR-001, T-ERR-002, T-ERR-003

## 5. 기능 분해 및 책임 할당
- `BoardValidator`
  - 입력: `board`
  - 출력: `ValidationResult`
  - 책임: 형식/범위/중복/합 제약 통합 검증
  - 부작용: 없음(순수 함수)
- `ConstraintChecker`
  - 입력: `state` 또는 `board`
  - 출력: 제약 통과 여부 및 오류코드
  - 책임: 단일 규칙 검사(분리된 함수군)
  - 부작용: 없음
- `StateTracker`
  - 입력: 현재 상태 + 배치 연산
  - 출력: 갱신 상태
  - 책임: `rowSums`, `colSums`, `usedNumbers` 관리
  - 부작용: 정책 선택(불변 복사 권장, 성능 필요 시 가역 변경 + 복구)
- `MagicSquareGenerator`
  - 입력: 옵션(탐색 순서, 해 개수 제한)
  - 출력: 유효 보드
  - 책임: 백트래킹 탐색, 가지치기 적용
  - 부작용: 내부 탐색 카운터(선택적)
- `ErrorCatalog`
  - 입력: 내부 검증 실패 이벤트
  - 출력: 표준 오류 코드/메시지
  - 책임: 테스트에서 안정적으로 assert 가능한 오류 계약 제공

테스트 가능성 강화 원칙:
- 규칙 함수는 단일 책임 + 순수 함수 유지
- 탐색 로직과 출력 포맷 분리
- 시간/랜덤 의존성 제거(결정적 탐색 순서)
- 공개 API의 계약(입력/출력/오류코드) 고정

## 6. 알고리즘/탐색 전략
비교 대상:
1. 순열 완전탐색(16! 기반)
   - 장점: 구현 단순
   - 단점: 비현실적 계산량, 테스트 실행 시간 과다
2. 백트래킹 + 제약 가지치기(선택)
   - 장점: 탐색 공간 대폭 축소, TDD로 규칙 단위 검증 용이
   - 단점: 상태 관리 복잡도 증가
3. 제약 전파(CSP) 강화형
   - 장점: 더 강한 pruning 가능
   - 단점: 초기 구현 부담 큼, 과설계 위험

선택 근거:
- TDD 친화성(규칙을 작은 테스트로 독립 검증 가능)
- 구현 난이도 대비 성능 균형
- 디버깅/실패 재현 용이성

성능 고려 및 가지치기 규칙:
- 부분 행/열 합이 34 초과 시 즉시 백트랙
- 행/열 마지막 셀 배치 시 합이 정확히 34가 아니면 중단
- 대각선 포함 셀 배치 시 대각선 상한/하한 검증
- 대칭성 제거(첫 셀/첫 행 고정 규칙)로 중복 탐색 축소
- 후보 선택 휴리스틱: 제약이 큰 셀 우선(MRV 유사 전략)

## 8. 리스크 및 대응 전략
- 리스크: 상태 변경 버그(백트래킹 복구 누락)
  - 대응: 상태 불변 객체 우선, 가역 변경 시 복구 테스트 추가
- 리스크: 테스트가 구현 세부사항에 과결합
  - 대응: 공개 계약(입출력/오류코드) 중심 assertion
- 리스크: 성능 저하로 테스트 지연
  - 대응: P0 경량 테스트와 성능 테스트 분리, pruning 규칙 점증 적용
- 리스크: 오류코드 난립
  - 대응: ErrorCatalog 단일 소스 유지, 코드 표준 문서화
- 테스트 취약 지점
  - 부분 상태 검증 누락, 대각선 경계조건, null/타입 혼합 입력
- 보강 전략
  - 속성 기반 테스트(범위/유일성 불변식), 파라미터라이즈드 테스트 확대

## 9. 최종 검증 계획
- 정확성 검증
  - 고정 샘플(정답/오답 보드) + 무작위 변형 보드로 `validate` 검증
  - `generate` 결과를 항상 `validate`로 이중 검증
- 회귀 테스트 전략
  - P0는 매 커밋 실행, P1/P2는 CI 파이프라인 단계 실행
  - 버그 발생 시 재현 테스트를 먼저 추가한 뒤 수정
- 문서/코드 일치성 점검
  - 요구사항 ID(RQ)별 테스트 ID 존재 여부 점검
  - 오류코드 카탈로그와 테스트 assert 값 동기화
  - 단계별 구현 계획의 완료 체크리스트와 실제 테스트 스위트 일치 검토

## 부록 B. 가정(Assumptions)
- 출력 정책은 "유효한 해 1개 반환"을 기본으로 한다.
- 입력 보드 검증 API와 생성 API는 분리되어 제공된다.
- 보드 좌표는 0-based 인덱스를 사용한다.
- 오류 처리 방식은 예외 throw보다 구조화된 오류코드 반환을 우선한다.
- 성능 목표는 "개발/CI 환경에서 실용적인 시간 내 단일 해 생성"이며, 절대 시간 SLA는 별도 비기능 요구사항에서 확정한다.
- 대칭 해(회전/반사)는 별도 중복 제거 요구가 없으면 허용한다.
